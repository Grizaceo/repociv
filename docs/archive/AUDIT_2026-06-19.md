## P1 — Arreglos / mejoras de alto valor

### P1.1 — Three.js (610 kB) se descarga eager en modo 2D canónico
`dist/index.html` hace `modulepreload` de `vendor-sugMAR3o.js` (**610 kB /
157 kB gzip**), que contiene Three.js entero (167 refs a `THREE.`,
`WebGLRenderer`, etc.), y el entry chunk lo referencia estáticamente.
- El render **canónico es Canvas 2D** (SCOPE §"branch paralela"); `src/three/*`
  solo se alcanza por `import()` dinámico (`renderMode.ts` →
  `await import('./ThreeMapRenderer.ts')`). Sin embargo el catch-all
  `manualChunks(id) → 'vendor'` (vite.config.ts) mete Three en el mismo chunk
  que una dep transitiva estática, lo que **vuelve eager los 610 kB**.
- Impacto: cada carga inicial (modo 2D, el 99% del uso) baja ~600 kB / 157 kB
  gzip de Three que no usa.
**Fix:** dar a Three su propio chunk para que quede lazy con `ThreeMapRenderer`:
```js
manualChunks(id) {
  if (!id.includes('/node_modules/')) return undefined;
  if (id.includes('/three/')) return 'vendor-three'; // ← aislar
  if (id.includes('/lucide/')) return 'vendor-icons';
  // ...resto igual
}
```
Verificar después que `vendor-three` **no** aparezca en los `modulepreload`
de `index.html`. Ahorro estimado: ~157 kB gzip en first paint 2D.

### P1.2 — Normal map de 3.3 MB en assets 3D
`public/assets/3d/terrain-normal-atlas-3d.png` = 3.3 MB (el archivo más
grande del repo). El budget de `check.sh` (≤6 MB para los 3 PNG) lo deja
pasar, pero un normal map a esa resolución es caro de descargar y de subir a
GPU. Total `public/` = 7.8 MB.
**Fix:** generarlo a menor resolución (la mayoría de detalle se pierde en
pantalla) y/o migrar las texturas 3D a **KTX2 + Basis** (compresión GPU,
`three` ya trae `KTX2Loader`). Es el camino correcto si el 3D va a quedarse.

### P1.3 — Drift de SCOPE: el 3D vive en `main`, no en branch paralela
`docs/SCOPE.md` §"Lo que sí pero en branch paralela": *"feat/3d-renderer —
branch independiente… el Canvas 2D sigue siendo el render canónico… no toca
trunk."* La realidad (`git log`): los últimos ~20 commits de `main` son 3D
(`feat(3d) P1-P5`, `feat(render) v28`…), **no existe branch `feat/3d-renderer`**,
y `NEXT_SESSION_GOAL.md` confirma *"Branch único: main, sin features"* con
force-push a `origin/main`.
- El propio SCOPE dice: *"Si entra al trunk igual, este documento queda
  desactualizado y deja de ser la fuente de verdad."* → está en esa situación.
**Fix (governance, no código):** decidir explícitamente una de dos y
actualizar `SCOPE.md`:
  (a) el 3D **es** ahora parte del trunk → reescribir §branch-paralela y
      añadir un **gate de paridad 2D/3D** real (ya existe
      `e2e/render-mode-parity.spec.ts`, formalizarlo como bloqueante); o
  (b) mantener 2D canónico → mover el trabajo 3D a una branch de verdad.

### P1.4 — Cobertura: umbral bajo + UI/renderers totalmente excluidos
`vite.config.ts` fija thresholds `lines: 55, branches: 70` y **excluye todo
`src/ui/**`, los 4 renderers, `localWorldManager`, `main.ts`** de coverage.
Es ~la mitad del front (incluido todo el código de UI/paneles) sin medición.
Razonable para Canvas/DOM puro, pero hay lógica testeable mezclada (los
`pickerLogic.ts`, `state.ts`, `markdown.ts` ya tienen tests → el patrón
"extraer lógica pura y testearla" funciona).
**Fix:** extraer la lógica no-DOM de los paneles más grandes
(`relationsPanel`, `observabilityPanel`, `onboardingPanel`) a módulos puros
testeables y subir el umbral de líneas a 65 gradualmente.

## Recomendaciones de funcionalidad (alineadas al SCOPE / dogfooding)

> No propongo features fuera de scope (multi-tenant, multiplayer, etc., ya
> vetadas). Estas sirven directamente las métricas de éxito del alpha.

1. **Telemetría de poda (lo que el SCOPE explícitamente pide).** SCOPE
   §"Roadmap de poda" necesita datos de qué paneles/endpoints se usan. Ya
   existe `_trackLayerToggle()` (layers.ts) y `server/endpoint_usage.py` →
   **conectarlos a un pequeño panel de "uso"** que liste paneles/endpoints con
   0 invocaciones en N días. Convierte la decisión de poda en algo basado en
   datos en vez de intuición. Alto ROI: es prerequisito de casi todo el resto
   del roadmap.

2. **Gate de paridad 2D↔3D.** Hay `e2e/render-mode-parity.spec.ts` pero el job
   visual de CI es *informative/non-blocking*. Definir un set mínimo de
   invariantes funcionales (mismo nº de ciudades/unidades/territorios en ambos
   modos) y hacerlo bloqueante. Resuelve la ambigüedad de P1.3 con un test, no
   con un doc.

3. **Budget de bundle en CI.** (ver P3) Un check de tamaño gzip del entry
   eager evita que vuelva a colarse algo como Three.js de 600 kB.

4. **Pin único de toolchain.** Fijar `ruff`/`eslint`/`prettier`/`vite` a
   versiones exactas y compartidas entre `requirements.txt`, `package.json`,
   pre-commit y CI, para que "verde local" == "verde CI" siempre (cierra P0.2).

5. **Pipeline de assets KTX2** (si el 3D se queda en trunk): generador →
   `.ktx2` comprimido en vez de PNG de 3.3 MB. Baja peso de red y memoria GPU,
   y el `check.sh` budget se vuelve holgado.
